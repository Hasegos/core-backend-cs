# ERD (Entity-Relationship Diagram)

ERD는 **객체-관계-다이어그램**으로, 데이터베이스의 구조를 한눈에 파악하기 위해 설계 단계에서 사용하는 시각적 도구입니다.

## ERD를 사용하는 이유

1. **시각화**: 복잡한 데이터 구조를 한눈에 파악할 수 있습니다.

2. **의사소통**: 개발자, 설계자, 사용자가 동일한 설계를 공유하고 이해도를 높입니다.

3. **설계 오류 방지**: 실제 DB 구축 전 중복을 제거하고 무결성을 확보할 수 있습니다.

4. **쿼리 최적화**: 테이블 간의 관계를 보며 효율적인 조인(Join) 전략을 세울 수 
있습니다.

## Entity (개체)

데이터를 저장하고자 하는 관리 대상(사람, 사물, 장소, 개념 등)입니다.

* **강한 엔티티 (Strong Entity)**: 독립적으로 존재하며, 고유한 식별자(PK)를 가집니다.

* **약한 엔티티 (Weak Entity)**: 독립적으로 존재할 수 없고 상위 엔티티에 의존합니다. (예: 노선관리 -> 노선별 정류장 순서)

## Attribute (속성)

엔티티를 구성하는 세부 항목이며, 반드시 데이터 타입을 명시해야 합니다.

| 속성 유형 | 설명 | 예시 |
| :--- | :--- | :--- |
| **Key Attribute** | 객체를 유일하게 식별하는 고유 값 | 사원번호, 학번, 주민번호 |
| **Composite Attribute** | 여러 세부 속성이 모여 구성됨 | 주소 (시+구+동), 성명 (성+이름) |
| **Multi-Valued Attribute** | 하나의 속성이 여러 값을 가짐 | 전화번호, 취미, 전공 |
| **Derived Attribute** | 다른 속성값으로부터 계산되어 도출됨 | 나이 (생일 기준), 총액 (수량*단가) |

## Relationship (관계)

엔티티 간의 논리적인 연결을 의미하며, **IE 표기법(까마귀발 표기법)**을 주로 사용합니다.

<img src="https://github.com/user-attachments/assets/9a8c1e35-c329-46e9-b551-c5cd4419c284" />

### 관계의 종류

1. **1:1 (One-to-One)**: 양쪽 데이터가 단 하나씩만 대응됩니다. (예: 사용자 - 프로필 정보)

<img width=400 src="https://github.com/user-attachments/assets/a3fefa2a-cd1c-4459-aa66-139598680a93" />

2. **1:N (One-to-Many)**: 한쪽의 데이터가 반대쪽 데이터 여러 개와 대응됩니다. (예: 부서 - 사원들)

<img width=400 src="https://github.com/user-attachments/assets/7485f849-46b7-47a5-9a36-a9455f8b99e7" />

3. **N:M (Many-to-Many)**: 양쪽 모두 여러 데이터와 대응됩니다. (예: 학생 - 수업)
    * *주의: 실제 DB 구현 시 N:M은 중간에 '연결 테이블'을 두어 1:N 관계로 풀어내야 합니다.*

## 한눈에 정리하기

ERD는 단순한 그림이 아니라 **데이터베이스의 설계도**입니다. 작은 프로젝트라도 사전에 ERD를 통해 데이터 무결성과 관계를 명확히 정의하면, 추후 유지보수 비용을 획기적으로 줄일 수 있습니다.
