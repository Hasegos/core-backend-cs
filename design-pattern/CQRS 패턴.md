# CQRS (Command Query Responsibility Segregation)

<strong>CQRS(명령 쿼리 책임 분리)</strong>는 데이터 저장소에 대한 **읽기(Query) 및 쓰기(Command) 작업을 별도의 모델로 분리**하는 디자인 패턴입니다.<br>
이를 통해 각 모델을 독립적으로 최적화하여 성능, 확장성 및 보안을 극대화할 수 있습니다.

## 왜 CQRS가 필요한가요? (Context & Problem)

기존의 단일 데이터 모델 기반 CRUD 아키텍처는 시스템이 커질수록 다음과 같은 한계에 부딪힙니다.

<img src="https://github.com/user-attachments/assets/d0eb3cbd-1dd4-41d9-b4cc-0d9ac81c1fea" />

* **잠금 경합 (Lock Contention)**: 동일 데이터에 대해 읽기와 쓰기가 동시에 발생하면 성능이 저하됩니다.

* **성능 불일치**: 읽기는 복잡한 조인이 필요하고, 쓰기는 단순 업데이트가 필요한 등 요구 사항이 서로 달라 최적화가 어렵습니다.

* **보안 관리의 어려움**: 읽기만 가능한 사용자에게 쓰기 모델의 속성까지 노출될 위험이 있습니다.

## 해결법: 명령과 쿼리의 분리

CQRS는 모든 작업을 크게 두 가지 책임으로 나눕니다.

1.  **명령 (Command)**: 시스템의 상태를 변경하는 작업 (Create, Update, Delete). 비즈니스 의도를 담아야 합니다<br>
(예: `ReservationStatus = 'Booked'` 대신 `MakeReservation`).

2.  **쿼리 (Query)**: 시스템의 상태를 조회하는 작업 (Read). 데이터를 변경하지 않으며, 결과로 **DTO**를 반환합니다.

## CQRS 구현 방식

### 1. 단일 저장소 모델 (Shared Database)

동일한 데이터베이스를 공유하지만, 코드 수준에서 읽기/쓰기 논리를 분리합니다. 

<img src="https://github.com/user-attachments/assets/08abcea8-ae26-43ab-9332-d14eb5f022fa" />

* **쓰기**: 복잡한 비즈니스 로직과 트랜잭션 무결성 보장에 집중.

* **읽기**: 조회 성능을 위해 최적화된 View나 DTO 반환에 집중.

### 2. 별도 저장소 모델 (Separate Databases)

읽기와 쓰기에 최적화된 서로 다른 DB 기술을 사용합니다. (예: 쓰기는 RDBMS, 읽기는 NoSQL)

* **동기화**: 쓰기 저장소에서 변경이 발생하면 <strong>이벤트(Event)</strong>를 게시하여 읽기 저장소를 업데이트합니다.

* **최종 일관성(Eventual Consistency)**: 두 DB가 실시간으로 완벽히 일치하지 않을 수 있으나, 시간이 지나면 결국 일치하게 됩니다.

## CQRS의 장점과 고려 사항

| 분류 | 주요 내용 |
| :--- | :--- |
| **이점 (Pros)** | **독립적 확장**: 읽기 부하가 높으면 읽기 서버만 증설 가능<br>**최적화**: 쿼리 성능 향상 (복잡한 Join 방지)<br>**관심사 분리**: 비즈니스 로직과 조회 로직의 결합 제거 |
| **단점 (Cons)** | **복잡도 증가**: 설계 및 구현 난이도가 높음<br>**일관성 문제**: 읽기 DB에 즉시 반영되지 않을 수 있음 (부실 데이터 가능성)<br>**메시징 관리**: 이벤트 브러커 및 동기화 실패 처리 필요 |

## 핵심 포인트

CQRS는 만능 해결책이 아닙니다. 단순한 CRUD 위주의 서비스에 도입하면 오히려 과도한 엔지니어링이 될 수 있습니다.<br>
**사용자가 많아 읽기 부하가 심각하거나, 비즈니스 로직이 매우 복잡하여 명령과 쿼리의 요구사항이 극명하게 갈리는 경우** 도입을 적극 검토해야 합니다.
