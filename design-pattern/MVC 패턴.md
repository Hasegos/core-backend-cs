## 왜 MVC 패턴이 등장했나요?

애플리케이션이 복잡해짐에 따라 <strong>"데이터 처리 로직"</strong>과 <strong>"화면 표시 로직"</strong>이 뒤섞여 유지보수가 매우 어려워졌습니다.<br>
화면을 바꾸기 위해 데이터 처리 코드까지 수정해야 하는 번거로움을 해결하고자, 두 로직을 독립적으로 분리하여 개발할 수 있는 MVC 패턴이 탄생했습니다.

## MVC 패턴이란?

MVC 패턴은 소프트웨어 설계 및 아키텍처를 위한 디자인 패턴 중 하나로, 응용 프로그램을 **Model, View, Controller** 세 가지 역할로 나누어 관리합니다.

### 구성 요소별 역할

* **Model (모델)**: 애플리케이션의 **핵심 데이터와 비즈니스 로직**을 담당합니다.<br>
데이터베이스와의 상호작용 및 데이터의 유효성 검사를 수행하며, 뷰나 컨트롤러에 의존하지 않는 독립적인 상태여야 합니다.

* **View (뷰)**: 사용자에게 보여지는 **UI(사용자 인터페이스)** 부분입니다.<br>
모델의 데이터를 화면에 표시하고, 사용자의 입력을 컨트롤러에 전달합니다.

* **Controller (컨트롤러)**: **사용자의 입력을 처리하고 애플리케이션의 흐름을 관리**합니다.<br>
뷰를 통해 전달된 요청을 분석하여 모델을 업데이트하고, 그 결과를 다시 뷰에 전달하여 화면을 갱신합니다.

## M-V-C의 상호작용 흐름

사용자가 앱에서 특정 작업을 수행할 때, 세 구성 요소는 다음과 같은 순서로 상호작용합니다.

1. **사용자 입력**: 사용자가 **View**를 통해 작업을 수행하면, **View**는 이를 감지하여 **Controller**에 전달합니다.

2. **요청 처리**: **Controller**는 입력을 분석해 적절한 **Model** 기능을 호출하여 데이터를 처리(검색/수정/저장)합니다.

3. **데이터 처리**: **Model**은 비즈니스 로직을 수행하고(필요 시 DB 연동), 작업 완료 결과를 **Controller**에 반환합니다.

4. **화면 갱신**: **Controller**는 모델로부터 받은 결과를 **View**에 전달하고, **View**는 이 데이터를 바탕으로 사용자 화면을 업데이트합니다.

## MVC 패턴의 5가지 핵심 설계 원칙

단순한 분리를 넘어, 더 견고한 시스템을 위해 다음의 원칙을 지켜야 합니다.

1. **Model은 View나 Controller를 몰라야 한다**: 데이터 로직은 UI 관련 코드에 의존해서는 안 됩니다.

2. **View는 Model의 정보만 가져올 뿐, 직접 수정할 수 없다**: 데이터 변경은 반드시 컨트롤러를 통해서만 이루어져야 합니다.

3. **View는 사용자 입력을 컨트롤러에 전달하는 역할만 수행한다**: 뷰 내부에서 비즈니스 로직을 처리하지 않습니다.

4. **Controller는 Model과 View를 연결하는 중재자 역할을 한다**: 양쪽의 흐름을 제어하되, 직접적인 비즈니스 로직을 담기보다는 모델을 호출하는 역할을 수행합니다.

5. **구성 요소 간의 결합도를 최소화한다**: 각 요소를 독립적으로 수정하고 테스트할 수 있도록 의존성을 줄여야 합니다.

## MVC 패턴의 장점

* **가독성과 유지보수 향상**: 역할 분담이 명확하여 코드를 이해하기 쉽고 수정이 용이합니다.

* **재사용성 증대**: 하나의 모델을 웹, 모바일 등 다양한 뷰에서 공통으로 사용할 수 있습니다.

* **병렬 개발 가능**: 디자이너(View)와 개발자(Model/Controller)가 서로 영향을 주지 않고 동시에 작업할 수 있습니다.

* **확장성**: 새로운 기능을 추가하거나 특정 부분을 교체할 때 독립적인 모듈로 개발되어 있어 대응이 빠릅니다.

## 한눈에 정리하기
MVC 패턴의 핵심은 <strong>"관심사의 분리(Separation of Concerns)"</strong>입니다.<br>
각 구성 요소가 자신의 역할에만 집중함으로써, 애플리케이션의 유연성을 높이고 지속 가능한 코드 구조를 유지할 수 있게 해줍니다.
