## 다형성

"<strong>역할과 구현</strong>"으로 세상을 구분하는 것이 다형성의 핵심입니다.

<img src="https://github.com/user-attachments/assets/2b18c694-5022-41ac-8738-c587cbfed692" />

> ### **핵심 개념**
> 자동차라는 **구현**이 바뀌더라도(K3 -> 테슬라), 운전자의 **역할**(운전할 수 있는 능력)은 바뀌지 않습니다.<br> 운전자는 자동차의 내부 구조를 몰라도 '운전 인터페이스'만 알면 자동차를 제어할 수 있습니다. 이것이 다형성이 주는 가장 큰 유연함입니다.

+  **클라이언트는 대상의 역할(인터페이스)만 알면 된다.**

+  **클라이언트는 구현 대상의 내부 구조를 몰라도 된다.**

+  **클라이언트는 구현 대상의 내부 구조가 변경되어도 영향을 받지 않는다.**

+  **클라이언트는 구현 대상 자체를 변경해도 영향을 받지 않는다.**

> ### **자바 언어에서의 적용**
> + **역할** = 인터페이스
> + **구현** = 인터페이스를 구현한 클래스, 구현 객체
> 
> 객체 설계 시 **역할(인터페이스)을 먼저 부여**하고, 그 역할을 수행하는 구현 객체를 만들어야 설계가 유연해집니다.

### 객체의 관계부터 정립

+  혼자 있는 객체는 없으며, 모든 객체는 서로 **협력 관계**를 가집니다.

+ <strong>클라이언트(요청)</strong>와 <strong>서버(응답)</strong>의 관계를 이해하는 것이 다형성 이해의 시작입니다.

### 자바의 오버라이딩(Overriding)

### 클라이언트 / 서버 구조의 의존성

### 다형성의 본질

+ 인터페이스를 구현한 객체 인스턴스를 **실행 시점에 유연하게 변경**할 수 있습니다.

+ 클라이언트를 변경하지 않고도 서버의 기능을 유연하게 확장할 수 있는 **확장 가능한 설계**의 핵심입니다.

## 스프링과 객체 지향

+ 스프링은 **다형성을 극대화**해서 이용할 수 있도록 돕는 프레임워크입니다.

+ **제어의 역전(IoC)**, **의존관계 주입(DI)** 기술을 통해 레고 블록을 조립하듯이 구현을 편리하게 변경할 수 있습니다.

## 좋은 객체 지향 설계의 5가지 원칙 (SOLID)

+ **SRP (단일 책임 원칙):** 한 클래스는 하나의 책임만 가져야 한다. (기준은 **변경** 시 파급효과)

+ **OCP (개방-폐쇄 원칙):** 확장에는 열려 있으나 변경에는 닫혀 있어야 한다.

+ **LSP (리스코프 치환 원칙):** 하위 타입은 상위 타입의 규약을 완전히 지켜야 한다.

+ **ISP (인터페이스 분리 원칙):** 범용 인터페이스 하나보다 특정 클라이언트를 위한 여러 인터페이스가 낫다.

+ **DIP (의존관계 역전 원칙):** 추상화에 의존해야지, 구체화에 의존하면 안 된다.

### OCP: 개방-폐쇄 원칙 (Open-Closed Principle)

<img src="https://github.com/user-attachments/assets/05ecd41b-17f7-4f8f-9f52-72cf67595f1b" />

+ **문제점:** 다형성을 사용해도 클라이언트가 구현 클래스를 직접 선택하면(`new ...`), 구현체 변경 시 클라이언트 코드 수정이 불가피하여 OCP가 깨집니다.

+ **해결법:** 객체를 생성하고 연관관계를 맺어주는 <strong>설정자(Spring Container)</strong>가 필요합니다.

### LSP: 리스코프 치환 원칙 (Liskov Substitution Principle)

+ 단순히 컴파일 성공을 넘어, 인터페이스의 의도(규약)를 지켜야 함을 의미합니다.
+ 예: 엑셀 기능을 뒤로 가게 구현하면 LSP 위반입니다. (느리더라도 앞으로 가야 함)

### DIP: 의존관계 역전 원칙 (Dependency Inversion Principle)

+ "<strong>구현체가 아닌 역할(Role)에 의존하라</strong>"는 원칙입니다.

+ **DIP 위반 사례:**

```java
  // MemberService가 인터페이스뿐만 아니라 구현체인 MemoryMemberRepository까지 의존하고 있음
  private MemberRepository memberRepository = new MemoryMemberRepository();
```

위 코드는 인터페이스와 동시에 구현체도 함께 의존하고 있어, 구현체 변경 시 클라이언트 코드까지 수정해야 하는 문제가 발생합니다.

### 요약

+ 객체 지향의 핵심은 다형성입니다.

+ 하지만 다형성만으로는 OCP, DIP를 완벽히 지키기 어렵습니다.

+ 스프링은 DI 컨테이너를 통해 다형성 + OCP, DIP를 가능하게 지원하여 부품 교체하듯 개발할 수 있게 합니다.

## 총정리

+ 모든 설계에서 역할과 구현을 분리해야 합니다.

+ 이상적으로는 모든 설계에 인터페이스를 부여하는 것이 좋습니다.

### 실무 고민

+ 인터페이스 도입에는 추상화라는 비용이 발생합니다.

+ 확장 가능성이 낮다면 구체 클래스를 먼저 사용하고, 향후 필요할 때 리팩터링을 통해 인터페이스를 도입하는 것도 실무적인 방법입니다.
