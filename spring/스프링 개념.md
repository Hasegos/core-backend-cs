## Framework

> **프레임워크란?** 어떠한 목적을 달성하기 위해 효율적으로 구조를 짜놓은 개발 틀(방식)을 의미합니다.

+ **프론트엔드 프레임워크/라이브러리:** React, Vue, Angular 등

+ **백엔드 프레임워크:** Spring Framework (Java 기반)

**Spring을 사용하는 이유:** JAVA로 웹 애플리케이션을 개발할 때 필요한 보안성, 유지보수 용이성, 안정성을 프레임워크 차원에서 강력하게 지원하기 때문입니다.

## 스프링

스프링의 핵심은 **스프링 컨테이너**입니다.<br>
컨테이너가 객체를 생성하고 관리하며, 우리는 이를 주입받아 사용합니다.

### Bean (빈)

+ 자바에서 `Car car = new Car();`로 생성하면 JVM이 관리하는 일반 객체입니다.

+ 스프링 컨테이너가 생성하고 관리하는 객체를 <strong>Bean(빈)</strong>이라고 부릅니다.

+ 스프링 컨테이너는 JVM의 특정 영역에서 빈들을 보관하고 있다가 필요할 때 꺼내어 줍니다.

### IoC (Inversion of Control, 제어의 역전)

+ 객체의 생성과 의존성 관리의 주도권이 개발자에게서 프레임워크(컨테이너)로 넘어가는 것을 말합니다.

### 주요 어노테이션

+ `@Component`: 해당 클래스를 스프링 빈으로 등록하라고 알려줍니다.

+ `@Configuration`: 설정을 담당하는 클래스임을 명시합니다.

+ `@ComponentScan`: `@Component`가 붙은 클래스들을 찾아 빈으로 등록합니다.

+ `@Autowired` / `@RequiredArgsConstructor`: 스프링 컨테이너에 등록된 빈을 필요한 곳에 **의존성 주입(DI)** 시켜줍니다.

> **의존성 주입(DI) 방식**
> + **생성자 주입:** 가장 권장되는 방식입니다. (`@RequiredArgsConstructor` 활용)
>
> + **필드 주입:** 사용은 간편하나 순환 참조 확인이 어렵고 외부에서 수정이 불가능해 권장되지 않습니다.

### 싱글톤 (Singleton)

+ 스프링은 기본적으로 빈을 **싱글톤**으로 생성합니다. 즉, 어디서든 같은 객체를 재활용하여 메모리 효율을 높입니다.

+ 만약 매번 새로운 인스턴스가 필요하다면 `@Scope("prototype")`을 사용합니다.

## Spring AOP
   
핵심 비즈니스 로직(예: 주문하기, 가입하기) 외에 공통적으로 발생하는 <strong>부가 기능(로깅, 트랜잭션, 보안 등)</strong>을 분리하여 관리하는 기술입니다.

+ **@Aspect:** AOP 기능을 정의하는 클래스에 사용합니다.

+ **Advice:** 언제 부가 기능을 실행할지 정의합니다. (`@Before`, `@After`, `@Around` 등)

### **예시**

```java
@Aspect
@Component
public class LoggingAspect {
    // drive() 메서드 실행 전 실행되는 Advice
    @Before("execution(* com.example.Car.drive(..))") 
    public void logBefore() {
        System.out.println("[Log] Car is about to drive");
    }
}
```

## Spring Security
애플리케이션의 <strong>인증(Authentication)</strong>과 <strong>권한(Authorization)</strong>을 담당하는 프레임워크입니다.

+ **인증(Authentication):** '이 유저가 누구인가?'를 확인 (로그인)

+ **권한(Authorization):** '이 유저가 이 페이지에 접근할 수 있는가?'를 확인 (관리자/일반유저 구분)

## JPA (Java Persistence API)

자바 객체와 데이터베이스(DB) 간의 언어 차이를 메워주는 **ORM(Object-Relational Mapping)** 기술 표준입니다.

+ SQL 쿼리를 직접 작성하는 대신, 자바 메서드(예: findAll(), save())를 통해 데이터를 조작할 수 있습니다.

+ 생산성이 매우 높으며 유지보수가 용이합니다.

## Spring Boot란

"<strong>기존 스프링의 복잡한 설정을 자동화하여 생산성을 극대화한 프레임워크</strong>"

+ 내장 WAS(Tomcat 등)를 포함하고 있어 별도의 서버 설치 없이 실행 가능합니다.

+ `starter` 의존성을 통해 필요한 라이브러리 설정을 자동으로 잡아줍니다.

## Session vs Cookie

웹 브라우저와 서버는 기본적으로 <strong>Stateless(상태 정보가 유지되지 않음)</strong>한 특성을 가집니다.<br>
이를 해결하기 위해 상태 유지 방식이 필요합니다.

+ **Session (세션):** 서버 메모리에 유저 정보를 저장합니다. 보안상 유리하지만 서버 리소스를 점유합니다.

+ **Cookie (쿠키):** 클라이언트(브라우저)에 저장되는 작은 정보 파일입니다. 세션 ID를 쿠키에 담아 주고받으며 인증 상태를 확인합니다.

### JWT (JSON Web Token)

REST API 방식에서 주로 사용하며, 서버에 상태를 저장하지 않는 **Stateless** 방식입니다.

입장권(Token) 자체에 유저 정보를 담고 있어 서버가 세션을 유지할 필요가 없습니다.

### Spring Security 세션 정책 (SessionCreationPolicy)

+ `ALWAYS`: 항상 세션 생성

+ `IF_REQUIRED`: 필요 시 생성 (기본값)

+ `NEVER`: 세션을 생성하지 않지만 이미 존재하면 사용

+ `STATELESS`: 세션을 전혀 생성하지도 사용하지도 않음 (JWT 방식에 적합)
